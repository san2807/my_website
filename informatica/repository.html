Repository (also called repository database)

* powercenter repository is a relational database managed by repository service.
* repository consists of database tables that stores metadata.
* metadata describes different types of objects, such as mappings & transformations (created using powercenter client tools)
* int service uses repository objects to ETL data.
* repository d/b tables contain instructions required to extract, transform and load data.
* repository also stores information such as permissions for users.

* all repository clients access repository d/b tables through repository service.
* each repository service manage a single repository.
* a repository service process is an instance of repository service that runs on a particular machine or node.
* repository service protects metadata in repository.



repository architecture


repository connectivity
* repository clients such as power center clients, int service, pmrep and infacmd connect to repository through rep service
* repository clients communicate with rep service through a specified port over a TCP/IP connection.
* configure port when u install rep service.
* as PC services can reside on multiple nodes in domain, rep service relies on another service called SERVICE manager to direct client requests to appropriate rep service process.

hw repository client connect to rep d/b
* rep client sends a rep connection request to master gateway node, which is entry point to domain.

understanding metadata
* rep stores metadata that describes how to extract, transform and load source and target data.

repository objects create through Designer
1. source definitions
2. target definitions
3. transformations
4. reusable transformations
5. mappings
6. mapplets
7. user-defined functions
8. multi-dimensional metadata

repository objects create through Workflow Manager
1. database connections
2. sessions
3. workflows
4. workflow tasks
5. worklets

repository object locks
* repository uses locks to prevent users from duplicating or overwriting work.
* rep service creates following types of locks on rep objects when u view/edit or run them in workflow
        * in-use lock
        * write-intent lock
        * execute lock

modifying dependent objects
* when u modify a child object, u may cause a parent object to become invalid.
* for e.g. if u modify a mapping by changing a port datatype to a incompatible datatype, session may become invalid.



version control
* if u have team-based development option, u can enable version control for repository
* a versioned rep stores multiple version of an object.
