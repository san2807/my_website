Facts



Granularity [1]
* first step in designing a fact table is to determine the granularity of the fact table.
* by granularity, v mean the lowest level of information that will be stored in the fact table.

Grain
* declaring grain is pivotal step in dimensional design.
* grain establishes what a single fact table row represents
* grain declaration becomes a binding contract on the desingn
* grain must be declared before choosing facts or dimensions

Type of facts [1]
* there r 3 types of facts
    * additive
    * semi-additive
    * non-additive

additive facts


conform facts
* if the same measurement appears in seprate fact tables, care must be taken to make sure technical definitions of the facts r identical if they
   r to be compared or computed togather [sanb1]

transaction facts tables
* a row in transaction fact table corresponds to a measurement event at a point in space & time [sanb1]

periodic snapshot fact tables
* row in periodic snapshot fact table summarizes many measurement events occuring over a period, such as day,week, or month. [sanb1]
* grain is the period not a month. [sanb1]

factless fact tables
* although most measurement events capture results, it is possible that event merely records a set of dimensional entities coming together
    at a moment in time. [sanb1]

aggregate fact tables or olap cubes
* r simple numeric roll ups of atomic fact table data built solely to accelerate query performance. [sanb1]

consolidated fact tables
* combine facts from multiple processes together into a single consolidated fact table if they can be expressed at the same grain. [sanb1]
